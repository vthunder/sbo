## North star

Evolve the SBO specs to enable **trustless apps that don't feel like "web3."** Flagship target: a **self-owned community** ("Discord/Reddit, but the community owns itself and members carry portable reputation across communities"). A community that issues attestations about identities subsumes web-of-trust and portable credentials.

Reasoning lens — three pillars each change taxes: (1) data-as-primitive, (2) local verifiability, (3) web-like legibility.

## Decisions locked (do not re-litigate)

- **Based design.** Writers post directly to the DA layer (Avail); no sequencer. All validity is computed by clients on replay. A sequencer was explicitly considered and **rejected** (over-solves; more trust than needed).
- **Identity = email.** Attributed by **browserid certificate + DNSSEC chain → one pinned DNS root KSK** (`/sys/trust/dns-root`). No durable user keys; key-rooted identities preserved only for roots + a future self-sovereign tier. browserid lives in `~/src/browserid-ng` and requires DNSSEC end-to-end (broker fallback also enforces it).
- **Privacy (Fork C) deferred** — Avail "Enigma" (TEE) is the planned answer later.
- **Expressiveness (Fork D) resisted** — keep validation declarative + LWW; no contract-like logic / computed constraints.
- **Local-first (Fork A) is the real foundation work** — needed for L5 (high-write content). Not yet specced.
- **Single umbrella version**: "SBO Protocol v0.5"; spec docs are no longer individually versioned (filenames have no version suffix; each carries a "Part of SBO Protocol v0.5" badge).

## What's in the specs now (Phase 2)

Commits this session: `620d35e` (v0.5 cleanup) → `969c422` (two-layer validity + DNSSEC attribution) → `bc6c949` (Identity rewrite) → `c8729c5` (Auth→Authorization) → `414468d` (reconcile rest).

Key concepts written down:
- **Two validity layers** (Core spec): L1 envelope validity (objective by deterministic replay); L2 attribution (made objective by self-authenticating DNSSEC evidence carried on chain). Authorization = signer resolves to owner via attribution + (indirect) ownership resolution.
- **DNSSEC-anchored attribution** (Authorization spec): `Auth-Cert` (browserid cert, envelope-as-assertion) + `Auth-Evidence` (DNSSEC chain, inline or `ref:` to a self-authenticating `dnssec.v1` object). Pinned anchors: `/sys/trust/dns-root`, `/sys/trust/brokers`. Inclusion-time clock = DA block timestamp (all cert/RRSIG/KSK windows must contain it). Full verification algorithm included.
- **Identity** (Identity spec): two kinds (email-rooted default `identity.email.v1` where controller = `Owner`; key-rooted `identity.v1` preserved). Tiers T0 (direct email) + T1 (community-as-IdP, pseudonymous, ban-the-person) normative; T2 (ZK-pseudonymous + nullifier) roadmap. `/sys/names/` resolution with grounding + cycle rules. Compact handles: `user@domain` (`@` reserved for browserid-attributable domains), `chain:appId/name` for no-domain repos, bare `name` in-repo.
- **State Commitment**: checkpoints are optional/performance; attribution durability does NOT depend on them.
- **Genesis**: Trust Anchors section; email-rooted user claims default; "two senses of domain" (on-chain root-of-trust domain vs DNSSEC-attributed provider domain).

Suite = 10 specs, no broken links, no version drift. All under `specs/`.

## Remaining spec work (the L3+ stack) — NOT started

Target app decomposes into layers (bottom-up): L0 substrate (done), L1 identity (done), L2 naming/URI (mostly done), **L3 attestation fabric**, **L4 community + governance**, **L5 content + local-first (Fork A)**, **L6 client + indexers**.

Recommended order:

1. **L3 — Attestation Specification (the keystone; START HERE).**
   - `attestation.v1`: a signed claim `{issuer, subject, type, value, issued_at, expires?, evidence?}`. Membership, roles, bans, vouches, badges, credentials are all instances.
   - **Decision already made:** use the **expiry-based / re-issuance** model (like browserid certs) rather than building strong revocation infra — revocation = a freshness problem, and expiry sidesteps the state-proof machinery. Strong revocation can come later.
   - Subject binds to the stable `/sys/names/<name>` anchor (survives email/controller changes).
   - Open design points to settle while writing: where attestations are stored (subject vs issuer namespace) and how scoring stays **off-chain** (indexers compute reputation as subjective views — keeps us out of Fork D).
   - After L3, web-of-trust + portable credentials already work — first shippable ecosystem primitive.

2. **L4 — Community Specification.** `community.v1` + membership lifecycle + moderation log, built on Policy v0.2 (roles/grants/`$owner` already exist). Governance stays non-expressive: admins = attestation-defined role + policy, NOT on-chain voting.

3. **L5 — Content + Fork A (the hard foundation).** Thin `post`/`comment`/`reaction` schemas + the **local-first write model**: HLC authoring clock, `Prev` hash-link, per-collection durability tiers (on-chain / batched / log-anchored), confirmed-vs-tip read semantics, anchoring into State Commitment. The on-chain clock primitive already pinned by the Authorization spec (block timestamp) is reused here.

4. **L6 — Indexers + reference client.** Local-first sync engine (extend `sbo-daemon`); indexers as off-chain competing views (feeds, search, reputation scoring) = the "open algorithm market"; reference community client wiring browserid + SBO.
